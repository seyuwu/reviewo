# Frontend Architecture

## Цель

Frontend должен быть максимально независимым от внутренней архитектуры backend.

Он никогда не должен знать:

- структуру базы данных;
- внутренние сервисы;
- бизнес-логику;
- способ хранения данных.

Frontend работает только с публичным API.

Главная задача frontend — отображать информацию и обеспечивать максимально простой пользовательский опыт.

---

# Основные принципы

## Компонентный подход

Любой экран собирается из небольших компонентов.

Например

Entity Page

↓

Header

↓

Rating Card

↓

Review List

↓

Statistics

↓

Related Entities

↓

Comments

Никаких страниц по 3000 строк.

---

## Feature-based архитектура

Структура строится вокруг возможностей пользователя.

Не так

pages/

utils/

services/

А так

src/

features/

entities/

ratings/

reviews/

search/

profile/

notifications/

settings/

Каждая feature полностью независима.

# Правила компонентов

Каждый компонент:

- выполняет одну задачу;
- имеет небольшой размер;
- легко тестируется;
- легко переиспользуется.

---

# Масштабирование

Добавление новой функции должно означать создание новой feature.

Существующий код должен изменяться минимально.

---

# Запрещается

- бизнес-логика внутри компонентов;
- запросы внутри JSX;
- большие страницы;
- большие компоненты;
- общие папки "helpers" со всем подряд.

# Независимость от Backend

Frontend никогда не должен зависеть от внутренней архитектуры backend.

Frontend не должен знать:

- какие существуют backend-модули;
- сколько сервисов участвует в обработке запроса;
- как устроена база данных;
- как данные агрегируются между доменами.

Frontend работает исключительно с публичными API-контрактами.

Например, для страницы объекта frontend получает единый ответ:

```ts
interface EntityPageResponse {
    entity: Entity;
    rating: Rating;
    trust: Trust;
    reviews: Review[];
}
```

При этом frontend не знает, были ли эти данные получены из одного модуля, нескольких модулей модульного монолита или из отдельных микросервисов.

Такой подход позволяет без изменений frontend постепенно переводить backend от модульного монолита к микросервисной архитектуре.
