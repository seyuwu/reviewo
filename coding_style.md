# Coding Style

## Цель

Цель данного документа — определить единый стиль разработки проекта.

Главный приоритет — не минимальное количество строк кода, а читаемость, простота сопровождения и возможность долгосрочного развития платформы.

Код пишется не только для компьютера, но и для других разработчиков, которые будут поддерживать проект спустя месяцы или годы.

---

# Главные принципы

## Простота важнее "умного" кода

Если существует выбор между:

- коротким, но сложным кодом;
    
- немного более длинным, но очевидным,
    

всегда выбирается второй вариант.

Код должен быть понятен без дополнительных объяснений.

---

## Один файл — одна ответственность

Каждый файл отвечает только за одну задачу.

Например:

- один React-компонент;
    
- один сервис;
    
- один Repository;
    
- один Controller;
    
- один Hook.
    

Не допускается объединение нескольких независимых сущностей в одном файле.

---

## Один модуль — одна бизнес-область

Каждый модуль отвечает только за одну часть системы.

Например:

- Ratings — только оценки;
    
- Reviews — только отзывы;
    
- Entities — только сущности;
    
- Search — только поиск.
    

Модули не должны содержать функциональность друг друга.

---

## Минимальная связанность

Код должен иметь минимальное количество зависимостей.

Чем меньше модуль знает о других частях системы, тем проще его изменять и масштабировать.

---

## DRY

Общая логика не должна дублироваться.

Если один и тот же код начинает повторяться — необходимо вынести его в отдельную функцию, сервис или общий модуль.

При этом не стоит преждевременно создавать абстракции ради нескольких строк кода.

---

## KISS

Предпочтение всегда отдается самому простому решению, которое полностью решает задачу.

Не следует реализовывать функциональность "на будущее", если она пока не нужна.

---

## YAGNI

Не добавлять код, который может пригодиться когда-нибудь.

Любая новая функциональность появляется только тогда, когда действительно необходима.

---

# Структура файлов

Каждый файл должен иметь понятную структуру.

Например:

```text
Imports

↓

Constants

↓

Types / Interfaces

↓

Main Class / Component / Service

↓

Private Functions

↓

Exports
```

Порядок должен быть одинаковым во всем проекте.

---

# Размер файлов

Следует стремиться к тому, чтобы файлы оставались небольшими.

Если файл становится слишком большим и начинает выполнять несколько задач, его необходимо разделить.

Точные ограничения не устанавливаются, однако большие файлы должны восприниматься как сигнал к рефакторингу, а не как норма.

---

# Именование

Используются понятные и полные названия.

Хорошо:

```text
EntityService

RatingRepository

calculateTrustScore()
```

Плохо:

```text
Service

Utils

Data

Manager

Helper

calc()
```

Название должно отражать назначение объекта.

---

# Компоненты

Каждый React-компонент отвечает за одну задачу.

Компоненты не должны содержать:

- запросы к API;
    
- сложную бизнес-логику;
    
- вычисления, не относящиеся к отображению.
    

Компонент получает данные и отображает их.

---

# Hooks

Каждый Hook отвечает за одну задачу.

Например:

- useEntity
    
- useRatings
    
- useReviews
    
- useSearch
    

Hooks не должны знать внутреннее устройство других модулей.

---

# Backend

Controller:

- принимает запрос;
    
- валидирует входные данные;
    
- вызывает Service;
    
- возвращает ответ.
    

Service:

- содержит бизнес-логику.
    

Repository:

- работает только с базой данных.
    

DTO:

- описывает входящие и исходящие данные.
    

---

# Ошибки

Не использовать "магические" строки.

Использовать собственные классы ошибок или централизованную обработку исключений.

Все ошибки должны иметь понятные сообщения и единый формат.

---

# Логирование

Логируются только действительно важные события.

Например:

- ошибки;
    
- создание объектов;
    
- модерация;
    
- подозрительная активность.
    

Не следует засорять лог информацией, не представляющей ценности.

---

# Комментарии

Комментарии используются только тогда, когда невозможно выразить смысл через понятный код.

Нельзя комментировать очевидные вещи.

Предпочтение всегда отдается понятным именам функций и переменных.

---

# Рефакторинг

Если изменение существующего кода требует большого количества условий (`if`, `switch`, вложенных проверок), необходимо задуматься о переработке архитектуры.

Рефакторинг является частью процесса разработки, а не отдельной задачей "на потом".

---

# Принцип эволюции архитектуры

При разработке платформы всегда предпочтение отдается решениям, которые позволяют системе постепенно развиваться.

Если существует выбор между:

- быстрым решением, которое усложнит развитие проекта;
    
- немного более сложным решением, которое позволит масштабироваться без переписывания системы,
    

всегда выбирается второй вариант.

Каждый новый модуль должен проектироваться так, будто однажды он станет самостоятельным сервисом.

При этом нельзя преждевременно усложнять архитектуру.

Проект развивается по принципу:

```text
Простой модульный монолит

↓

Четкие границы доменов

↓

Постепенное выделение отдельных модулей

↓

Полноценная микросервисная архитектура
```

Эволюция системы должна происходить постепенно, без полной переработки существующего кода.

---

# Правило новой функциональности

Любая новая возможность проходит одинаковый жизненный цикл:

```text
Идея

↓

RFC

↓

Backend Module

↓

API

↓

Frontend Feature

↓

Browser Extension

↓

Тестирование

↓

Релиз
```

Это позволяет сохранять единый подход к разработке независимо от размера проекта.

---

# Главный принцип проекта

Каждое архитектурное и техническое решение должно отвечать на вопрос:

> **Сделает ли это платформу проще, понятнее и удобнее для развития через несколько лет?**

Если ответ отрицательный — решение должно быть пересмотрено.
