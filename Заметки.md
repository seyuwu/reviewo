# Universal Rating Platform (рабочее название)

# Общая концепция

## Идея проекта

Universal Rating Platform — это универсальная платформа общественного мнения, позволяющая оценивать практически любую существующую сущность.

Главный принцип проекта:

> **Если что-то существует — это можно оценить.**

Сервис не ограничивается сайтами, фильмами или товарами. Любая сущность может получить собственную страницу с рейтингом, отзывами и статистикой.

Например:

- сайты;
    
- страницы сайтов;
    
- видео;
    
- каналы;
    
- AI-сервисы;
    
- GitHub-репозитории;
    
- статьи;
    
- приложения;
    
- книги;
    
- фильмы;
    
- игры;
    
- компании;
    
- преподаватели;
    
- рестораны;
    
- университеты;
    
- мероприятия;
    
- устройства;
    
- автомобили;
    
- любые реальные объекты.
    

Интернет становится лишь первым большим источником автоматически создаваемых сущностей.

## Философия проекта

Проект строится вокруг принципа, что оценивать должно быть так же просто, как искать информацию в поисковой системе.

Пользователь не должен задумываться о категориях, структуре данных, внутренней архитектуре платформы или процессе создания нового объекта.

Если нужный объект уже существует — он должен находиться за несколько секунд.

Если его еще нет — его создание должно занимать не более нескольких простых действий.

Главная задача платформы — сделать процесс оценки любого объекта максимально быстрым, интуитивным и естественным.

---

# Главная цель

Создать **слой общественного мнения поверх интернета и реального мира**.

Сегодня поисковые системы отвечают:

> Что это такое?

А проект должен отвечать:

> Что об этом думают люди?

Причем ответ должен появляться за несколько секунд.

---

# Основные принципы проекта

## Максимальная простота

Любое действие должно выполняться максимально быстро.

Пользователь не должен изучать интерфейс.

Не должен понимать архитектуру.

Не должен создавать сущности вручную.

Он должен просто захотеть что-то оценить.

---

## Сущности

Любой объект представляет собой Entity.

Каждая сущность имеет:

- уникальный ID;
    
- название;
    
- тип;
    
- родительскую сущность;
    
- дочерние сущности;
    
- ссылки;
    
- изображения;
    
- рейтинг;
    
- надежность рейтинга;
    
- отзывы;
    
- историю;
    
- статистику.
    

Все сущности объединяются в единое дерево.

Например:

Internet

→ YouTube

→ MrBeast

→ Видео

→ Комментарий

или

Internet

→ Amazon

→ Seller

→ Product

или

Internet

→ GitHub

→ Organization

→ Repository

→ Issue

Такой подход позволяет получать рейтинг любого уровня.

---

## Канонические сущности

Одна и та же сущность может иметь множество ссылок.

Например:

- ссылки с UTM;
    
- сокращенные ссылки;
    
- таймкоды;
    
- различные параметры.
    

Все они автоматически объединяются в одну сущность.

Пользователь всегда попадает на одну страницу объекта.

# Взаимодействие пользователя

# UX-принципы

При разработке любых новых функций главным приоритетом остается простота использования.

Любое действие пользователя должно занимать минимальное количество шагов.

Идеальный сценарий выглядит следующим образом:

Поиск

↓

Найден объект

↓

Оценка

↓

Готово

Пользователь никогда не должен проходить длинные мастера создания объектов, заполнять большие формы или изучать сложный интерфейс.

Если какое-либо действие можно автоматизировать, оно должно автоматизироваться.

Главная задача интерфейса — не заставлять пользователя думать о системе, а позволять ему сосредоточиться только на своем мнении.

## Главная страница

Главным элементом сайта становится одна строка поиска.

```
Что вы хотите оценить?
```

Пока пользователь печатает, отображаются найденные сущности.

Если нужной сущности нет, появляется пункт

```
Создать новую страницу
```

После нажатия открывается максимально простая форма.

Заполняются только самые необходимые поля:

- название;
    
- категория;
    
- ссылка (необязательно).
    

После создания пользователь автоматически переходит на страницу сущности и становится первым человеком, который может поставить ей оценку.

Кроме обычной оценки отображается показатель доверия.

Он рассчитывается автоматически.

Учитываются:

- количество пользователей;
    
- репутация пользователей;

- согласованность оценок;

- признаки накрутки.
    

Таким образом пользователь понимает не только оценку, но и степень доверия к ней.

# Архитектурные принципы

Проект изначально разрабатывается как **модульный монолит**, который в будущем можно постепенно преобразовать в полноценную микросервисную архитектуру без переписывания бизнес-логики.

Главная цель — получить преимущества монолита на этапе MVP (простая разработка, быстрое внесение изменений, отсутствие сложной инфраструктуры), но при этом сразу заложить архитектуру, позволяющую безболезненно масштабироваться по мере роста проекта.

Каждый модуль представляет собой отдельный домен предметной области и взаимодействует с другими модулями только через публичные интерфейсы. Ни один модуль не должен знать внутреннюю реализацию другого.

Архитектура проектируется не вокруг кода, а вокруг бизнес-доменов.

---

# Архитектура проекта

```text
apps/

├── web/                 # Next.js
├── extension/           # Browser Extension
├── api/                 # NestJS (модульный монолит)

packages/

├── ui/                  # Общие UI-компоненты
├── shared/              # Общие утилиты
├── types/               # Общие типы
├── config/              # Конфигурация
├── eslint/
└── tsconfig/
```

---

# Структура backend

Внутри backend каждый модуль полностью независим.

```text
apps/api/src/modules/

├── auth/
├── users/
├── entities/
├── ratings/
├── reviews/
├── trust/
├── search/
├── moderation/
├── notifications/
├── recommendation/
└── common/
```

Каждый модуль является отдельной бизнес-областью.

Например:

- Auth отвечает только за авторизацию.
    
- Users — только за пользователей.
    
- Entities — только за сущности и дерево объектов.
    
- Ratings — только за оценки.
    
- Reviews — только за отзывы.
    
- Trust — только за расчет надежности рейтинга.
    
- Search — только за поиск.
    
- Recommendation — только за рекомендации.
    
- Notifications — только за уведомления.

Проект изначально строится как модульный монолит.

Каждый модуль проектируется как отдельный бизнес-домен с четкими границами ответственности и публичными интерфейсами взаимодействия.

Это позволяет быстро разрабатывать MVP, сохраняя возможность постепенно выделять отдельные модули в полноценные микросервисы без переписывания существующей бизнес-логики.

---

# Внутренняя структура каждого модуля

Каждый модуль имеет одинаковую структуру.

Например:

```text
ratings/

├── controllers/
├── services/
├── repositories/
├── entities/
├── dto/
├── interfaces/
├── events/
├── tests/
└── ratings.module.ts
```

Благодаря этому структура проекта остается одинаковой независимо от количества модулей.

---

# Взаимодействие модулей

Модули не должны обращаться друг к другу напрямую.

Например:

❌ Плохо

```text
RatingModule

↓

EntityRepository

↓

UserRepository

↓

ReviewRepository
```

Такой код создает сильную связанность и усложняет переход к микросервисам.


Правильный подход:

```text
RatingModule

↓

EntityPort (публичный интерфейс)

↓

EntityModule
```

Каждый модуль предоставляет только публичный API (порт), а остальные модули ничего не знают о его внутреннем устройстве.

Это позволяет в будущем заменить локальный вызов метода на HTTP, gRPC или очередь сообщений практически без изменения бизнес-логики.

# Принципы масштабирования

Архитектура проекта должна позволять постепенно масштабировать платформу без полного переписывания системы.

Для этого соблюдаются следующие принципы:

- каждый модуль имеет одну область ответственности;
- каждый модуль обладает собственным публичным API;
- каждый модуль может быть вынесен в отдельный сервис;
- бизнес-логика не должна дублироваться между модулями;
- взаимодействие между модулями происходит только через публичные интерфейсы;
- внутреннее устройство модуля не должно быть известно другим модулям.

Такой подход позволяет эволюционировать от модульного монолита к полноценной микросервисной архитектуре постепенно, выделяя отдельные домены по мере роста нагрузки и команды разработки.

# API

Frontend никогда не обращается напрямую к отдельным модулям.

Существует единая точка входа.

```text
Web / Extension

↓

API

↓

Modules

├── Auth
├── Entities
├── Ratings
├── Reviews
├── Search
└── ...
```

Если в будущем модули будут вынесены в отдельные микросервисы, frontend продолжит работать через тот же API без каких-либо изменений.

---

# Переход к микросервисам

Архитектура изначально строится таким образом, чтобы любой модуль можно было вынести отдельно.

Например:

```text
2026

NestJS Modular Monolith

↓

Выделяем Search

↓

Выделяем Notifications

↓

Выделяем Recommendation

↓

Выделяем Ratings

↓

Выделяем Reviews

↓

Полноценная микросервисная архитектура
```

Каждый сервис переносится отдельно без переписывания всего проекта.

Именно поэтому MVP строится значительно быстрее, но не становится тупиковой архитектурой.

---

# Правила разработки

1. Каждый новый функционал реализуется в виде отдельного модуля.
    
2. Модули взаимодействуют только через публичные интерфейсы.
    
3. Бизнес-логика не должна дублироваться.
    
4. Любой модуль можно вынести в отдельный сервис без изменения остальных модулей.
    
5. UI состоит из небольших переиспользуемых компонентов.
    
6. Внутри каждого модуля используется единая структура каталогов.
    
7. Код полностью типизирован.
    
8. Каждый модуль покрывается тестами независимо от остальных.
    
9. Пользователь никогда не должен видеть сложность архитектуры — для него любое действие должно выполняться максимально быстро и интуитивно.
    
10. Архитектура должна оставаться понятной даже спустя годы разработки и при работе нескольких команд одновременно.
11. Любое архитектурное решение должно приниматься с учетом долгосрочного развития платформы. Если новая функция усложняет систему без существенной пользы для пользователя, предпочтение всегда отдается более простому и расширяемому решению.
